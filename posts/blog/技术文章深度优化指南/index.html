本文是技术文章优化的可复用提示词模板，适用于 content/posts/tech/ 目录下的深度学习、编程、数学等技术文章。使用前替换 {{文件路径}} 和 {{系列名}}。

安全执行声明（请先阅读）

  1. 长文章处理：若文章超过 10,000 字，请先确认全文已完整读取，再开始输出计划。
  2. 执行节奏：我选择「直接执行」模式，输出优化计划后无需等待确认，立即逐项实施。
  3. 附录使用：附录中的类比、模板仅供参考，不贴切时请自由发挥，不要生搬硬套。
  4. 文章的格式受 \posts\blog\技术文章格式规约.md 制约。

1. 强制工作流程（不可跳步）#

在使用 AI 优化技术文章时，必须遵守以下工作流程，避免"改了半天方向不对"的问题：

  1. 通读全文：识别硬伤（数值缺失、逻辑断裂、概念含糊）
  2. 输出优化计划：按 P0>P1>P2 列出待改项，等待确认后再动手
  3. 逐项实施：每完成一个 ## 大节，立即执行该节末尾的锚点自检（见第 6 节）
  4. 最终验收：全部改完后，按第 7 节格式输出验收报告

2. 核心目标（三问原则）#

让读者能扎实回答三个问题：

  • 是什么？（定义清晰）
  • 为什么？（设计选择的动机）
  • 怎么用？（有可验证的示例）

一篇好的技术文章应该让读者真正理解每个概念，而不只是"知道有这么个东西"。全文要有一条清晰的逻辑主线，章节之间有自然的过渡和衔接。

3. P0 级要求（硬性指标，缺一不可）#

3.1 关键算法必须有"手算数值示例"#

适用对象：反向传播、注意力机制、梯度下降、贝叶斯推断、矩阵特征值等。

设计铁律：

  • 用简单整数或一位小数，展示输入→中间步骤→输出→结果验证（如"损失从 0.18 降到 0.004"）
  • 示例后紧跟"解读"段落，把数字翻译成直觉

常见算法的数值示例清单：

算法示例要求
反向传播2层网络，前向传播→梯度计算→参数更新→验证损失下降
注意力机制3个简化向量，Q·K→Softmax→加权求和
梯度下降简单函数（如 $y=x^2$），3-5步迭代表格
贝叶斯定理“疾病检测"场景，先验→似然→后验
矩阵运算2×2 或 3×3 矩阵的手算

3.2 逻辑不能断裂#

  • 章节衔接：每章开头用"问题引入”（如前章遗留问题）+“本节目标”
  • 前向引用：若代码或概念依赖后文，必须明确标注（如"DataLoader 将在 2.3 节详解"）
  • 后向呼应：引入新方案时，点明它解决了前文的哪个痛点

3.3 概念解释必须准确无误#

禁止含糊其辞或直接搬用原文。若涉及数学，必须给出符号说明和直觉翻译（一句话概括公式在"做什么"）。

4. P1 级要求（强烈建议，显著提升深度）#

4.1 “为什么"分析框架（问题→方案→代价）#

针对每个架构、训练、数学设计选择，套用此模板：

  • 问题：不这样做会怎样？
  • 方案：这样做如何解决？
  • 代价：引入了什么新问题？
领域必须回答的"为什么”
架构设计为什么用 ReLU 而不是 Sigmoid？为什么用 CNN 处理图像？
训练技巧为什么需要 Dropout？为什么用 Adam 而不是 SGD？
数学公式为什么 Softmax 要除以 $\sqrt{d_k}$？为什么用交叉熵而不是 MSE？
工程选择为什么用混合精度训练？为什么需要 KV Cache？

4.2 核心概念配备类比（附边界说明）#

好的类比必须满足三个条件：

  • 贴切：类比的核心机制与原概念一致
  • 具体：有具体的场景和角色
  • 有边界：说明类比在哪里失效

4.3 代码注释解释"意图"#

代码示例不仅是"能跑"，更要"能教"：

# 差：只写操作，不解释为什么
optimizer.zero_grad()
loss.backward()
optimizer.step()

# 好：每步解释意图和类比
optimizer.zero_grad()   # 清零梯度（PyTorch默认累加，不清零会膨胀）
loss.backward()         # 自动计算每个参数的梯度（反向传播）
optimizer.step()        # 沿梯度反方向走一步（下山）

超过 30 行的代码块必须按"定义/准备/训练/评估"拆分，每段加注释说明在做什么。

5. P2 级要求（锦上添花）#

  • 每个 ## 大节末尾添加模块小结表格（你学到了什么→为什么重要）
  • 文章开头添加知识依赖图（Mermaid flowchart）
  • 首尾添加跨文章链接（{{< relref >}}），格式：**上一篇**：[标题]({{< relref "文件.md" >}}) — 一句话概括

可视化辅助#

类型适用场景示例
SVG 图表数学概念（梯度下降轨迹、过拟合曲线、几何直觉）等高线图、激活函数对比图
Mermaid 流程图架构和流程（知识依赖、模型架构、训练循环）CNN 数据流、Seq2Seq 架构
表格对比和总结（方法对比、选择指南、模块小结）LSTM vs GRU、优化器选择

6. 执行锚点（每改完一个 ## 大节，立即自检）#

  • 本节核心算法有无手算数值示例？（P0）
  • 本节设计选择有无"问题→方案→代价"分析？（P1）
  • 本节有无与上下章的衔接句？（P0）
  • 本节代码有无意图注释？超过30行是否已拆分？（P1）
  • 本节末尾有无模块小结表格？（P2）

7. 最终验收报告（修改完成后必须输出）#

按以下模板输出验收报告：

验收报告：
- P0 完成：数值示例 [X/关键算法数]；逻辑衔接 [X/章]；前向/后向引用 [X/处]
- P1 完成：为什么分析 [X/处]；类比 [X/个]；公式直觉翻译 [X/个]；代码注释 [X/块]
- P2 完成：模块小结 [X/节]；知识依赖图 [X/张]；跨文章链接 [X/个]
- 未完成项及理由：___

8. 参考附录（按需查阅，非强制执行）#

8.1 常用类比库（附边界说明）#

概念类比边界说明
梯度下降下山（海拔=损失，方向=梯度）真实地形高维且有局部最优，下山是低维直觉
反向传播工厂流水线责任追溯实际链式法则更复杂，且存在梯度消失或爆炸
Dropout期末考试随机缺席测试时不用缺席（需缩放），仅解释训练阶段
CNN手电筒照局部区域实际有步长、填充、多通道，类比忽略细节
RNN 梯度消失传话游戏（信息逐级失真）LSTM/GRU 通过门控缓解，但未完全消除长程依赖
注意力同声传译（回头看原文）实际有自注意力和交叉注意力，计算复杂度 $O(n^2)$
L1 正则化恒定摩擦（容易归零）仅对线性模型严格成立，深层网络更复杂
L2 正则化弹簧（权重越大拉力越大）对偏置项通常不惩罚，需注意例外

8.2 常见反面模式速查#

反面模式为什么有问题正确做法
公式后直接跳下一个公式读者还没消化就进入下一个公式后必须跟"直觉翻译"段落
代码片段无上下文读者不知道上下文，无法运行给完整端到端示例，或说明前置依赖
类比无限延伸不匹配的部分会误导读者在脚注中明确类比起作用的边界
所有内容同等详略重点不突出，读者抓不住主线核心创新详写，历史演进用表格简列
只列公式不给数值示例读者看得懂符号，但没有直觉公式后必须跟一个手算示例
一节堆砌太多概念认知负荷过重，什么都记不住每节聚焦1-2个核心概念

8.3 风格一致性模板（本系列统一）#

章节开头句式：

前面我们学习了 X，但还存在 Y 问题。本节将介绍 Z 来解决它。

章节结尾句式：

下一节我们将学习 W，它是在 Z 基础上的进一步扩展。

模块小结表格格式：

你学到了什么为什么重要
概念A它解决了X问题，让模型具备了Y能力

代码注释格式：

# 操作名（解释：为什么做，解决什么问题）

类比引入格式：

可以把 X 想象成 Y：...（注：类比的边界在于...）

模块小结#

你学到了什么为什么重要
强制工作流程避免"改了半天方向不对"，先读后改
三问原则每个知识点回答"是什么、为什么、怎么用"
P0/P1/P2 优先级有限时间做最重要的事
手算数值示例最有效的理解方式，让读者能用计算器验证
“为什么"分析框架问题→方案→代价三层分析
类比库与边界说明贴切、具体、有边界的类比才能真正帮助理解
反面模式识别避免"看起来好但实际有问题"的写法
